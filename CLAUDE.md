# Claude Code 项目工作规则

## 项目总原则

这是一个 Codroid 机器人控制 UI 项目。项目使用 PySide6 开发，包含登录页、主窗口、顶部功能标签栏、页面路由、左侧机器人运动控制抽屉、底部全局命令栏、状态栏、TCP 通信、UDP/CRI 实时数据、机器人运动控制、Jog 点动、moveTo RunTo 控制等功能。

任何修改都必须以当前源码为准，不要凭空假设接口或架构。

## github 同步规则

每次用户确认完成一阶段，就要把代码push到github上

## plan.md 同步规则

每次收到用户提出的新需求、功能变更、UI 设计变更、接口行为变更、线程/通信规则变更、项目架构变更时，必须先检查 `plan.md` 是否已经记录该内容。

如果 `plan.md` 没有记录，或者记录与当前需求不一致，必须同步更新 `plan.md`。

`plan.md` 的用途是项目交接文档，必须让任何新的开发者或 AI 拿到后都能理解：
1. 项目做了什么；
2. 使用什么技术栈；
3. 项目目录结构；
4. UI 架构如何组织；
5. TCP/UDP/CRI 通信如何组织；
6. QThread + Signal/Slot 线程规则；
7. 机器人协议接口如何调用；
8. 当前已完成什么；
9. 当前未完成什么；
10. 后续开发注意事项；
11. 哪些文件负责哪些功能。

## 修改流程要求

执行任何代码修改前，必须遵守以下流程：

1. 先阅读相关源码；
2. 先阅读 `plan.md`；
3. 判断用户当前需求是否已经写入 `plan.md`；
4. 如果没有写入，先补充或更新 `plan.md`；
5. 再修改源码；
6. 修改完成后，在回复中说明：
   - 修改了哪些源码文件；
   - 是否更新了 `plan.md`；
   - `plan.md` 更新了哪些内容；
   - 需要用户测试什么。

## 严禁行为

禁止只改源码、不更新 `plan.md`。
禁止在不了解现有源码的情况下重写架构。
禁止一次性扫描或改动无关文件。
禁止在未说明影响范围的情况下大改通信、线程或 UI 主结构。
禁止保留旧逻辑导致两套入口同时存在，例如同一功能出现两套按钮、两套 signal、两套数据流。

## UI 当前设计基准

主界面采用：
- 顶部功能标签栏；
- 中间 `QStackedWidget` 页面区域；
- 左侧可收缩运动控制抽屉；
- 底部全局命令栏；
- 状态栏。

左侧运动控制抽屉当前设计：
- 收起状态只显示“运动”按钮；
- 展开后显示机器人运动控制面板；
- 上方预留 3D 模型显示区域；
- 中间为“关节点动 / 坐标系点动”切换；
- 下面显示 J1~J6 或 X/Y/Z/Rx/Ry/Rz 点动按钮；
- 2x2 预设位按钮：零点、安全点、蜡烛位、打包位；
- 速度滑条默认 70%。

## moveTo 控制规则

`Robot/moveTo` 不是点击一次执行到底，而是按住保持型 RunTo 控制：

按下预设位按钮：
- 发送 `Robot/moveTo`；
- `db.type` 为：
  - `0` = Home / 零点；
  - `1` = 安全点；
  - `2` = 蜡烛位；
  - `3` = 打包位；
- 响应后每 0.5 秒发送一次 `Robot/moveToHeartbeat`。

松开预设位按钮：
- 立即停止 `moveToHeartbeat`；
- 发送 `Robot/moveTo`，其中 `db.type = -1`。

不得把 moveTo 实现成单击式按钮。

## Jog 控制规则

Jog 点动也是按住式：
- pressed：发送 `Robot/jog` 并启动 `Robot/jogHeartbeat`；
- released：停止心跳并发送 `Robot/stopJog`。

## 速度规则

连接成功后默认下发速度 70%。

速度滑条变化后，需要同时设置：
- 手动模式速度；
- 自动模式速度。

远程模式使用自动模式速度。

## 线程规则

PySide6 中所有 `QTcpSocket`、`QUdpSocket`、`QTimer` 的创建、启动、停止、销毁必须发生在其所属线程。

跨线程调用必须使用 Signal → @Slot。

禁止从 UI 线程直接操作 TCP/UDP 子线程里的 socket/timer。
禁止从 TCP/UDP 子线程直接更新 UI。

## 用户/客户手动修改源码后的处理规则

客户或用户有时会在 Claude Code 之外手动修改源码。

当用户说：
- “我改了某个文件”
- “客户改了某个文件”
- “我手动改了 xxx.py”
- “你看看我改的这个文件”
- “这个文件我动过了”
- “客户那边改了源码”

Claude Code 必须立即执行以下流程：

1. 暂停继续修改源码；
2. 先读取用户指定的文件；
3. 如果项目使用 Git，必须优先查看：
   - `git status`
   - `git diff -- <指定文件>`
4. 对比当前文件与原计划/原架构是否冲突；
5. 总结用户/客户手动修改了什么；
6. 判断这些修改是否需要同步写入 `plan.md`；
7. 如果需要，先更新 `plan.md`；
8. 再继续后续开发任务。

禁止在没有查看用户/客户手动修改内容的情况下继续改同一个文件。

禁止直接覆盖用户/客户手动修改。

如果发现用户/客户修改与当前实现方案冲突，必须先向用户说明冲突点，并给出合并建议，不要擅自回滚。
