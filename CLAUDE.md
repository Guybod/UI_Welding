# Claude Code 项目工作规则

## 0. 项目总原则

这是一个 Codroid 机器人控制 UI 项目。

项目使用 **Python + PySide6** 开发，包含但不限于：

- 登录页；
- 主窗口；
- 顶部功能标签栏；
- 页面路由；
- 左侧可收缩机器人运动控制抽屉；
- 底部全局命令栏；
- 状态栏；
- TCP 通信；
- UDP / CRI 实时数据；
- 机器人连接管理；
- 机器人运动控制；
- Jog 点动；
- moveTo / RunTo 按住式控制；
- 后续扩展功能页：焊接、写字、上传、HTTP、WebSocket 等。

任何修改都必须以**当前源码**和**当前 `plan.md`** 为准。

禁止凭空假设接口、架构、类名、信号名、线程模型或通信协议。

---

## 1. 默认工作模式

除非用户明确要求立即修改，否则 Claude Code 默认必须先进入 **plan mode**。

涉及以下任意内容时，必须先规划、再执行：

- UI 架构调整；
- 页面结构调整；
- 登录流程调整；
- 主窗口结构调整；
- 连接管理调整；
- TCP / UDP / CRI 通信调整；
- QThread / QTimer / Socket 生命周期调整；
- Jog / moveTo / RunTo 运动控制调整；
- 状态栏、底部命令栏、侧边抽屉调整；
- 新增功能页；
- 重构；
- 修改 `plan.md`；
- 多文件联动修改。

默认流程：

1. 阅读用户需求；
2. 阅读 `plan.md`；
3. 阅读相关源码；
4. 判断是否需要调用 subagent；
5. 由 main 汇总 subagent 结论；
6. 输出分阶段、分 part 的实施计划；
7. 等待用户确认；
8. 用户确认后再修改源码；
9. 修改完成后说明修改文件、影响范围、测试方法；
10. 等用户测试通过后再进入下一 part。

未经用户确认，不允许直接执行大范围代码修改。

## 3. GitHub 同步规则

每次用户确认完成一个阶段后，必须将代码 push 到 GitHub。

执行 push 前必须先检查：

```bash
git status
git diff
```

如果发现有用户或客户手动修改内容，必须先向用户说明，不得直接覆盖。

推荐流程：

1. `git status`
2. `git diff`
3. 确认本阶段修改范围；
4. 提交 commit；
5. push 到 GitHub；
6. 回复用户 commit 信息和 push 结果。

禁止在用户未确认阶段完成前擅自 push。

## 4. plan.md 同步规则

每次收到用户提出的新需求、功能变更、UI 设计变更、接口行为变更、线程/通信规则变更、项目架构变更时，必须先检查 `plan.md` 是否已经记录该内容。

如果 `plan.md` 没有记录，或者记录与当前需求不一致，必须同步更新 `plan.md`。

`plan.md` 是项目交接文档，必须让任何新的开发者或 AI 拿到后都能理解：

1. 项目做了什么；
2. 使用什么技术栈；
3. 项目目录结构；
4. UI 架构如何组织；
5. TCP / UDP / CRI 通信如何组织；
6. QThread + Signal / Slot 线程规则；
7. 机器人协议接口如何调用；
8. 当前已完成什么；
9. 当前未完成什么；
10. 后续开发注意事项；
11. 哪些文件负责哪些功能；
12. 当前功能的验收标准；
13. 当前禁止修改或暂缓修改的范围。

### 4.1 plan.md 更新要求

更新 `plan.md` 时必须：

- 保持原有结构清晰；
- 不得删除仍然有效的历史约束；
- 不得把临时想法写成已完成事项；
- 已完成、进行中、未完成必须区分；
- 对通信、线程、UI 主结构的变更必须写清楚影响范围；
- 对用户明确要求的规则必须原样保留其语义。

## 5. 修改流程要求

执行任何代码修改前，必须遵守以下流程：

1. 先阅读相关源码；
2. 先阅读 `plan.md`；
3. 判断用户当前需求是否已经写入 `plan.md`；
4. 如果没有写入，先补充或更新 `plan.md`；
5. 判断是否需要调用 subagent；
6. 如需要，先调用对应 subagent 审查；
7. main 汇总审查结论；
8. 输出修改计划；
9. 等待用户确认；
10. 再修改源码；
11. 修改完成后，在回复中说明：
    - 修改了哪些源码文件；
    - 是否更新了 `plan.md`；
    - `plan.md` 更新了哪些内容；
    - 是否调用了 subagent；
    - 每个 subagent 的核心结论；
    - 需要用户测试什么；
    - 有哪些风险；
    - 如何回滚。

## 6. 严禁行为

禁止只改源码、不更新 `plan.md`。

禁止在不了解现有源码的情况下重写架构。

禁止一次性扫描或改动无关文件。

禁止在未说明影响范围的情况下大改通信、线程或 UI 主结构。

禁止保留旧逻辑导致两套入口同时存在，例如同一功能出现两套按钮、两套 signal、两套数据流。

禁止 UI 层直接操作底层 socket。

禁止子线程直接操作 QWidget。

禁止从 UI 线程直接操作 TCP / UDP 子线程里的 socket / timer。

禁止从 TCP / UDP 子线程直接更新 UI。

禁止把所有业务逻辑塞进 MainWindow、LoginPage 或单个 ConnectionManager。

禁止在未确认线程生命周期的情况下创建新的 QThread / QTimer / Socket。

禁止在未阅读现有协议封装的情况下新增一套重复通信接口。

禁止在用户未确认前执行大范围重构。

禁止直接覆盖用户或客户手动修改的源码。

- 顶部功能标签栏；
- 中间 `QStackedWidget` 页面区域；
- 左侧可收缩运动控制抽屉；
- 底部全局命令栏；
- 状态栏。

左侧运动控制抽屉当前设计：

- 收起状态只显示“运动”按钮；
- 展开后显示机器人运动控制面板；
- 上方预留 3D 模型显示区域；
- 中间为“关节点动 / 坐标系点动”切换；
- 下面显示 J1~J6 或 X/Y/Z/Rx/Ry/Rz 点动按钮；
- 2x2 预设位按钮：
  - 零点；
  - 安全点；
  - 蜡烛位；
  - 打包位；
- 速度滑条默认 70%。

### 7.1 后续功能页扩展基准

后续功能页应支持逐步扩展，包括但不限于：

- 主页；
- 焊接；
- 写字；
- 上传；
- 调试；
- 设置；
- HTTP 功能；
- WebSocket 功能。

顶部功能标签栏如果页面过多，应支持横向滚动或合理的页面管理方式。

全局机器人控制区域与具体功能页区域必须职责分离：

- 左侧运动控制抽屉：全局机器人运动控制；
- 底部全局命令栏：全局启动、停止、暂停、恢复、模式切换等；
- 功能页内部：只放该功能专属操作，例如焊接页中的送气、送丝、退丝、试运行等。

## 8. moveTo 控制规则

`Robot/moveTo` 不是点击一次执行到底，而是按住保持型 RunTo 控制。

按下预设位按钮：

- 发送 `Robot/moveTo`；
- `db.type` 为：
  - `0` = Home / 零点；
  - `1` = 安全点；
  - `2` = 蜡烛位；
  - `3` = 打包位；
- 响应后每 0.5 秒发送一次 `Robot/moveToHeartbeat`。

松开预设位按钮：

- 立即停止 `moveToHeartbeat`；
- 发送 `Robot/moveTo`，其中 `db.type = -1`。

不得把 moveTo 实现成单击式按钮。

### 8.1 moveTo 实现注意事项

- moveTo heartbeat 必须有明确启动和停止条件。
- 按钮 released、窗口关闭、断线、重连、页面销毁时，都必须确保 heartbeat 停止。
- 不得出现松开按钮后仍继续发送 moveTo heartbeat 的情况。
- moveTo 与 Jog 不得共用不清晰的状态变量。
- moveTo 停止逻辑不得依赖 UI 是否仍然可见。

---

## 9. Jog 控制规则

Jog 点动也是按住式：

- pressed：发送 `Robot/jog` 并启动 `Robot/jogHeartbeat`；
- released：停止心跳并发送 `Robot/stopJog`。

### 9.1 Jog 实现注意事项

- Jog heartbeat 必须有明确启动和停止条件。
- 按钮 released、窗口关闭、断线、重连、页面销毁时，都必须确保 Jog 停止。
- 不得在子线程直接更新 Jog 按钮状态。
- 不得把 Jog 实现成单击式运动。
- Jog 与 moveTo 的状态必须隔离。

## 10. 速度规则

连接成功后默认下发速度 70%。

速度滑条变化后，需要同时设置：

- 手动模式速度；
- 自动模式速度。

远程模式使用自动模式速度。

### 10.1 速度实现注意事项

- UI 显示值、内部缓存值、机器人实际下发值必须一致。
- 速度设置失败时必须有日志或状态提示。
- 重连成功后应恢复当前 UI 速度设置。
- 默认速度 70% 不得被硬编码散落在多个文件里，建议集中管理。

## 11. 线程规则

PySide6 中所有 `QTcpSocket`、`QUdpSocket`、`QTimer` 的创建、启动、停止、销毁必须发生在其所属线程。

跨线程调用必须使用 Signal → `@Slot`。

禁止从 UI 线程直接操作 TCP / UDP 子线程里的 socket / timer。

禁止从 TCP / UDP 子线程直接更新 UI。

### 11.1 QThread 生命周期规则

所有 QThread 必须具备明确生命周期：

1. 创建；
2. moveToThread；
3. start；
4. 正常运行；
5. 请求停止；
6. 停止 timer / socket / worker；
7. thread.quit；
8. thread.wait；
9. deleteLater 或安全释放。

禁止出现：

```text
QThread: Destroyed while thread is still running
```

如果涉及线程销毁，必须重点检查：

- worker 是否仍在运行；
- timer 是否已停止；
- socket 是否已关闭；
- 信号是否重复连接；
- thread 是否 wait 完成；
- 应用退出时是否统一清理。

### 11.2 跨线程通信规则

- UI → Worker：必须通过 Signal 调用 worker 的 Slot。
- Worker → UI：必须通过 Signal 返回主线程。
- Worker 内部状态可用 Lock / Queue / 状态快照保护。
- UI 不得直接读取正在被子线程写入的共享对象。
- 高频 UDP 数据不得直接驱动高频 UI 刷新。
- UI 应以较低频率读取状态快照或接收节流后的状态信号。

## 12. TCP / UDP / CRI 通信规则

### 12.1 连接成功后的默认动作

连接成功后必须执行必要初始化，包括：

1. 设置默认速度 70%；
2. 如果需要 CRI 数据推送，先发送 `CRI/StopDataPush`；
3. 再发送 `CRI/StartDataPush`；
4. 启动或恢复必要的状态订阅；
5. 启动必要 heartbeat；
6. 更新 UI 连接状态。

### 12.2 重连成功后的恢复动作

掉线后优先自动重连。

重连成功后必须重新执行连接恢复逻辑：

1. 停止旧 CRI 推送；
2. 重新启动 CRI 推送；
3. 恢复状态订阅；
4. 恢复速度设置；
5. 刷新 UI 状态；
6. 清除或更新重连弹窗。

### 12.3 StartDataPush 规则

`StartDataPush` 比较特殊。

每次连接成功或重连成功后，如果要启动数据推送，必须先发送一次：

```json
{
  "ty": "CRI/StopDataPush",
  "db": null
}
```

然后再发送：

```json
{
  "ty": "CRI/StartDataPush",
  "db": {
    "ip": "<local_ip>",
    "port": "<udp_port>",
    "duration": 20,
    "mask": "<mask>",
    "highPercision": true
  }
}
```

### 12.4 本地 IP 和 UDP 端口规则

登录界面必须包含：

- 机器人 IP；
- 本地 IP 下拉选择；
- UDP 端口。

本地 IP 下拉选择要求：

- 自动枚举本机网卡；
- 显示网卡名称；
- 显示当前 IPv4 地址；
- 由客户手动选择使用哪个本地 IP。

UDP 端口规则：

- 默认从 `10000` 到 `65535` 中随机选择一个；
- 必须避免明显已占用端口；
- 用户可在必要时修改。

### 12.5 断线与重连弹窗规则

掉线后优先自动重连。

重连弹窗要求：

- 显示当前正在重连；
- 显示重连次数；
- 显示重连耗时或最近一次重连时间；
- 只保留“返回主页面”按钮；
- 不提供容易误操作的多余按钮；
- 重连成功后自动关闭或更新状态。

## 13. 机器人状态缓存规则

机器人状态不得直接散落在 UI 控件中。

建议使用统一状态模型或状态快照，至少包含：

- connected；
- reconnecting；
- enabled；
- moving；
- alarm；
- estop；
- mode；
- robot_ip；
- local_ip；
- udp_port；
- joint position；
- cartesian pose；
- speed；
- last update time；
- last error。

### 13.1 状态流规则

推荐状态流：

```text
TCP / UDP / CRI Worker
        ↓
Robot State Cache / Snapshot
        ↓
Application Service / ConnectionManager
        ↓
Signal
        ↓
UI
```

禁止状态流：

```text
UI 直接读写 socket
UI 直接解析 UDP 高频数据
Worker 直接操作 QWidget
多个页面各自维护一套机器人连接状态
```

## 14. 用户/客户手动修改源码后的处理规则

客户或用户有时会在 Claude Code 之外手动修改源码。

当用户说：

- “我改了某个文件”
- “客户改了某个文件”
- “我手动改了 xxx.py”
- “你看看我改的这个文件”
- “这个文件我动过了”
- “客户那边改了源码”

Claude Code 必须立即执行以下流程：

1. 暂停继续修改源码；
2. 先读取用户指定的文件；
3. 如果项目使用 Git，必须优先查看：
   - `git status`
   - `git diff -- <指定文件>`
4. 对比当前文件与原计划/原架构是否冲突；
5. 总结用户/客户手动修改了什么；
6. 判断这些修改是否需要同步写入 `plan.md`；
7. 如果需要，先更新 `plan.md`；
8. 再继续后续开发任务。

禁止在没有查看用户/客户手动修改内容的情况下继续改同一个文件。

禁止直接覆盖用户/客户手动修改。

如果发现用户/客户修改与当前实现方案冲突，必须先向用户说明冲突点，并给出合并建议，不要擅自回滚。

## 17. 最小改动原则

除非用户明确要求重构，否则优先采用最小改动方案。

最小改动不等于临时补丁。

最小改动必须满足：

- 不破坏现有功能；
- 不引入第二套入口；
- 不绕过现有架构；
- 不增加不可维护的全局变量；
- 不恶化线程安全；
- 不恶化通信稳定性；
- 能被写入 `plan.md`；
- 能被测试验证。

## 20. 当前项目关键约束摘要

- 项目是 PySide6 机器人控制 UI。
- `plan.md` 是核心交接文档，必须保持同步。
- 连接成功后默认速度 70%。
- 速度变化后必须同时设置手动速度和自动速度。
- 远程模式使用自动速度。
- `Robot/moveTo` 是按住保持型 RunTo 控制，不是单击式。
- Jog 是按住式点动，不是单击式。
- `StartDataPush` 前必须先 `StopDataPush`。
- 掉线后优先自动重连。
- 重连成功后必须重新恢复 CRI 数据推送和必要订阅。
- 本地 IP 由登录页下拉选择，显示网卡名称和 IPv4。
- UDP 端口默认随机选择 `10000~65535`。
- 线程中 socket / timer 必须在所属线程创建、启动、停止、销毁。
- 跨线程必须 Signal → Slot。
- UI 不得直接操作 socket。
- 子线程不得直接操作 QWidget。
- 不得直接覆盖用户/客户手动修改。
- 每个阶段完成并经用户确认后，才 push 到 GitHub。
